Plan 02 · Wireframe · ADR-0012

Loom Mobile

Your agents ask, your phone answers. Observe the fleet from anywhere and resolve the things that block it — instead of work stalling until you walk back to the laptop.

Shaped by the 2026-07-15 grill. The hero is answering the Agent that is blocked on you — an Approval waits indefinitely, so a push reaches you and the work resumes. Driving the fleet (send / read) carries nearly the whole risk budget and is second-class: it is not what justifies the app. A Clearance sits beside it but is mostly a local fix — it dies with its caller in ~2 min (rule 3.5).

Clearance inbox — mostly a local fix
Loom holding a command, pending your go/no-go. Actor ≠ decider, so it is a real authorization boundary — and each card names its Host (rule 7: an unattributed prompt is unanswerable once two are paired).

But be honest about the wall clock: loom spawn runs through the Agent's own tool call, which times out at 120s. You will rarely answer one of these from a phone — it's dead before the push lands. So a Clearance is bound to caller liveness, not a timer, and is withdrawn — not denied — when the caller gives up.

Its real value is local: these guardrails are window.confirm today, which blocks the webview thread and freezes every Pane's rendering until someone walks over. P0b ships that with no phone at all — which is now its main justification.

Attention — this is the hero
An Agent blocked on its own stdin, reporting it needs you. It waits indefinitely — so the push reaches you at lunch, you answer, and the work resumes. That is the whole product: your agents ask, your phone answers.

Answering is a send (y), and it must cost one tap — so the Confirmation fires only when the text is destructive (reusing isDestructiveCommand). A Confirmation stops no attacker anyway; its whole value is typo protection, so it belongs where typos hurt and nowhere else.

Still a separate list from Clearances: an Agent raises an Approval about its own work; Loom raises a Clearance about a command. Don't merge the inboxes.

Pane detail — swipe between Panes
Swipes within the Workspace, in layout-tree order, so "next" on the phone is the Pane next to it on the laptop. Dead Panes included — Otis's exit code is exactly what you want when an agent died while you were out. The strip is the affordance (a bare gesture is undiscoverable) and doubles as state: you can see Cleo needs you from two Panes away.

Swipe forces the Read Window. A tap per read would cost six taps to scan this Workspace — so approve resolves once into a 15-minute window (banner, top). Not an unlock: send still Confirms every time, every read is still audited, and the duration is a security parameter — it's how long a stolen unlocked phone reads freely.

Note the terminal wraps. It must: read returns lines wider than a phone, and a horizontally-panning terminal makes the swipe ambiguous. Decide that before the ANSI renderer gets written — there's no xterm.js on React Native, so it's built from scratch either way.

4 · Absent by design

rule 3 · deny

An op earns a screen by having a use case — not by existing and looking survivable.

✕ spawn
A silent-RCE primitive. No screen, so it is absent rather than gated. If it's ever wanted it costs a table row and a screen — the right ceremony.
✕ broadcast
The human bar was removed 2026-06-25 as unused, and its single-Workspace scope is why. A phone would rebuild that flaw on a worse keyboard.
✕ gate.set
Would let a Device drop the gate then send — walking the guardrail in two calls. Closed, not prompted.
✕ status · attention
Setters, despite the names. Granting them would let a Device rewrite labels and clear borders fleet-wide. Judge an op by its payload.
✕ prefetch on swipe
Not an op — a temptation. Each prefetched neighbour is a real read: scrollback pulled you never looked at, and pressure on the rate limiter. Fetch on settle.
✕ cross-Workspace swipe
Swipe stays inside the Workspace — the hierarchy is exactly two levels, and Broadcast doesn't cross either. Use the fleet list to change Workspace.

Revoke — enforced at the Bridge
Deletes the stored public key. The next handshake fails at the Host — so revocation needs neither the Relay's honesty nor its reachability. A compromised Relay can break your uptime; it can never resurrect a revoked Device.
