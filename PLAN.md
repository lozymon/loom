# Loom — Terminal Control Room

> **Historical build log.** This records the M0–M11 milestone plan as built, in its original "terminal multiplexer" framing. Loom's identity has since moved to an **agent-first developer environment** — see [ADR-0008](docs/adr/0008-agents-first-class-via-self-report.md), [ADR-0011](docs/adr/0011-heuristic-output-observer.md), and [AGENT_FIRST_PLAN.md](AGENT_FIRST_PLAN.md). The engine described here is unchanged; only the product framing moved.

> A Linux-first desktop **agent-first developer environment** built on real terminals — many real PTYs at once in resizable split grids and a left workspace rail (a GUI tmux/Terminator substrate), tuned for driving and observing a fleet of CLI agents. Built on Tauri 2 + SolidJS + xterm.js, with PTYs managed in Rust.

## Context
A "control room" of live terminals. Each pane is a real pseudo-terminal running any command — a shell, `claude`, a dev server, `tail -f`, `vim`. You split panes into a resizable grid, group them into Workspaces on a left rail, and (the headline trick) **broadcast one input to many panes at once** to launch and prompt a whole fleet in lockstep. The engine is generic — a pane is any PTY — but the product leads with agents (first-class since ADR-0008). This doc is the executable, incremental build plan; it front-loads the riskiest assumption (high-throughput PTY output rendering smoothly through Tauri IPC into WebKitGTK). *(Update 2026-06-25: the human broadcast bar was later removed as unused; one-to-many fan-out is now agent-driven only, via the `loom broadcast` control-bus command — see docs/ASSESSMENT.md.)*

**Locked-in stack:**
- **Shell:** Tauri 2 (Rust + system WebKitGTK webview) — small bundles, fast on Linux.
- **Frontend:** SolidJS + Vite + TypeScript. Terminal rendering via **xterm.js** (`@xterm/xterm` 5.x).
- **PTYs:** Rust **`portable-pty`** (wezterm crate) — one PTY + child process per pane.
- **Persistence:** local JSON config (serde) for workspaces/layouts. SQLite only if/when searchable scrollback or session logging is wanted (deferred).

**Environment:** Node v24, Rust 1.96/cargo, Linux Mint 22.3 (apt). Project lives at `/home/lozymon/code/loom/`.

## Architecture
```
SolidJS webview ── invoke(commands) ───────────────────────► Rust (Tauri core)
   xterm.js per pane                                          │  owns: PtyManager,
   layout tree (splits)   ◄── Channel<bytes> (PTY output) ────┤  one PTY + reader thread
   workspace rail (left)                                      │  per pane, resize, kill,
   broadcast input                                            │  workspace JSON persistence
        │                                                     ▼
        └── invoke pty_write / pty_resize ──────►  N× PTY masters → child procs
                                                   ($SHELL, claude, anything)
```
**One PtyManager owning `HashMap<PaneId, PtySession>`** — each session is a PTY master, its child process, and a dedicated reader thread that streams output bytes to the webview over a **Tauri `Channel`** (not events — channels are the fast binary-streaming path). Input (keystrokes) and control (resize/kill) flow the other way as cheap Tauri commands. This is symmetric to the old "one sidecar, N sessions" idea but the unit of isolation is now an OS PTY, which the kernel isolates for free.

### Language boundary (Rust does the core now)
The old (pre-pivot, SDK-based) plan kept Rust to glue; this pivot moves the engine into Rust because PTYs are an OS concern. Split:
- **Rust — the PTY engine.** `PtyManager`: spawn a PTY with a requested command/cwd/env, run the child, pump master→Channel on a reader thread, accept writes/resizes/kills, clean up on child exit. Plus workspace JSON load/save and packaging path resolution. Still no *product* logic — layout math, focus, broadcast routing all live in TS.
- **TypeScript — all UX/state.** SolidJS owns the layout tree (split geometry, drag-resize), the workspace rail, focus/zoom, the broadcast router, and binding each xterm instance to its pane's Channel. xterm.js owns ANSI parsing, scrollback, selection, rendering.
- **Shared types** (`PaneId`, `PaneSpec`, `LayoutNode`, command/payload shapes) live in one TS module (`src/ipc/protocol.ts`); the Rust command signatures mirror it.

### Data flow & performance (the make-or-break)
Terminal output is bursty and high-volume (`yes`, `cat bigfile`, a noisy build). The risk is saturating the Tauri IPC bridge or the WebKitGTK main thread. Mitigations, designed in from M0:
- **Output:** each PTY reader thread reads into a buffer and **coalesces** on three dimensions, flushing on whichever trips first — a ~8–16ms **tick**, a 32–64KB **buffer size**, and a **per-flush byte cap** (a flooding pane drops intermediate already-scrolled-off bytes rather than shipping all of them; xterm only needs to reach the correct screen state). Caps both frames-per-second and bytes-per-flush per pane regardless of throughput. Send raw bytes; avoid per-line JSON.
- **Transport:** **Tauri `Channel` first** (idiomatic, no open port/auth surface), but M0 is a spike — Tauri IPC may base64-encode channel payloads into WebKitGTK. Named fallback if Channel can't meet the M0 bar: a **localhost WebSocket** served by Rust streaming true binary frames. Callers talk to a thin `ptyClient` abstraction so swapping transport never touches them. See [ADR-0003](docs/adr/0003-output-transport-channel-first-websocket-fallback.md).
- **Render:** feed chunks straight into `term.write()` — xterm.js has its own internal write buffer and async flush, so don't pre-split into per-token DOM work. **Renderer: canvas-leading** (`@xterm/addon-canvas`), **DOM** as the ultra-safe fallback, **WebGL experimental opt-in only** (never default — most likely to fail silently on WebKitGTK). M0 picks the winner empirically and records it. See [ADR-0006](docs/adr/0006-canvas-renderer-not-webgl.md).
- **Input:** keystrokes → `pty_write(paneId, bytes)` command. Low volume, fine as plain commands. Never round-trip output through commands.
- **Backpressure:** bounded per-pane buffers; if a pane floods, drop-coalesce rather than queue unboundedly. A hidden Workspace's panes keep their PTY alive but throttle their flush rate.
- **Limits:** **no hard cap** on panes/Workspaces — bound cost by *visibility*, not count. A collapsed pane or hidden-Workspace pane isn't in the DOM (Solid `<Show>`) so it never *paints*; Rust keeps its output coalescing at a throttled rate so the xterm is current on reveal. The wizard's "12" is a creation-convenience ceiling, not a system limit. If a real WebKitGTK ceiling emerges in testing, **soft-warn** (don't forbid) and record the finding.

### Layout model
The **workspace rail** is a left vertical list (not top tabs). Each entry is a **Workspace** — a named container carrying a `working folder`, a **layout tree**, and a live terminal-count badge, with a close ✕. Switching the rail swaps the whole grid; PTYs in hidden Workspaces stay alive. Hierarchy is exactly two levels (Workspace → Pane); there is no intermediate "window" layer. Each tree leaf is a **named Pane** bound to a PTY.
```ts
type LayoutNode =
  | { kind: 'leaf'; paneId: PaneId }
  | { kind: 'split'; dir: 'row' | 'col'; ratio: number; a: LayoutNode; b: LayoutNode };
type Workspace = { id: string; name: string; cwd: string; tree: LayoutNode; panes: Record<PaneId, PaneSpec> };
```
- Splitting a leaf replaces it with a `split` whose children are the old leaf + a new leaf (new PTY). The new pane runs the **default `$SHELL`** (a split is "another terminal here," not a clone of the focused pane's command); default split direction is **split-right** (vertical divider, side-by-side). Closing a leaf collapses its parent split, promoting the sibling.
- Draggable dividers mutate `ratio` (0–1). Each pane measures its pixel box with `@xterm/addon-fit` → derives cols/rows → `pty_resize(paneId, cols, rows)`. Resize is debounced.
- Recursive SolidJS `<LayoutNode>` component renders splits as flex containers with a draggable gutter between `a` and `b`. Workspaces lazily mount panes but keep PTYs running when hidden.
- **Each pane has a title bar:** a **name** + per-pane controls (collapse / expand-split / maximize-zoom / close). The focused pane shows a focus ring. The name is auto-assigned from a curated pool of short distinct human names (Faye, Cleo, Wade…), **unique within the Workspace** (overflow → `Pane N`), purely a label (identity is the `PaneId`), **persisted** in `PaneSpec.title`, and **user-renamable** (double-click). Distinct names matter precisely because a fleet of panes often runs the *same* command (`claude` ×6) and would otherwise be indistinguishable.
- **Input routing:** **click-to-focus**, exactly one focused Pane; all keystrokes route to its PTY via `pty_write`. Mouse wheel scrolls that Pane's xterm scrollback locally (reaches the PTY only when the inner app enables mouse mode). App shortcuts occupy **only the `Ctrl+Shift+<key>` namespace** (`C/V` copy/paste, `D` split, `W` close, `←/→/↑/↓` move focus, `T` new Workspace…); every other combo — plain `Ctrl+C` (SIGINT), `Ctrl+B`, arrows, function keys — passes through untouched. No prefix key. See [ADR-0005](docs/adr/0005-ctrl-shift-shortcut-namespace.md).

### New-workspace wizard (the **+** on the rail)
Clicking **+** on the workspace rail opens a 3-step setup wizard (**Start → Layout → Agents**):
1. **Start** — pick the **working folder** (where every terminal in this workspace starts). Surfaces **Recent workspaces** (folder + remembered terminal count) and **Presets** (saved configs) for one-click relaunch.
2. **Layout** — a row of **grid-preset tiles**: 1, 2, 4, 6, 8, 10, 12 terminals. Tapping a tile builds a **width-biased** balanced grid via `buildBalancedTree(n)` — `rows = floor(√n)`, `cols = ceil(n/rows)`, so tiles stay as wide as possible: 2→1×2, 4→2×2, 6→2×3, 8→2×4, 10→2×5, 12→3×4 (rows×cols). The freeform splitter still works after launch; this is the fast on-ramp, not a separate mode.
3. **Agents** *(optional)* — assign a launch command per pane (default `$SHELL`; one-click presets like `claude`). Exit via **"Open without AI"** (all panes = plain shells) or **"Next: Add AI agents"** then launch.
A workspace can be saved as a **Preset** (name + folder + layout + per-pane commands) for one-click relaunch from step 1.

### Shared protocol (Tauri commands + Channel)
- **Commands (TS→Rust):** `pty_spawn(spec: PaneSpec) -> PaneId` (returns id + opens its output Channel), `pty_write(paneId, bytes)`, `pty_resize(paneId, cols, rows)`, `pty_kill(paneId)`, `pty_list()`, `workspace_load() -> Workspace`, `workspace_save(ws)`.
- **PaneSpec:** `{ command?: string; args?: string[]; cwd?: string; env?: Record<string,string>; cols: number; rows: number }`. A plain Pane spawns the user's **login + interactive `$SHELL`** (e.g. `bash -l`); a Pane *with* a command runs it via `$SHELL -lc "<command>"` (not a direct `execvp`) so PATH + shell init load — see [ADR-0004](docs/adr/0004-launch-via-login-interactive-shell.md). Env = app base env + per-pane `env` + injected `TERM=xterm-256color`, `COLORTERM=truecolor`. Command-exit → Dead pane.
- **Channel (Rust→TS):** raw output bytes per pane, coalesced. Plus a small `pty_exit` event/channel frame carrying exit code when the child dies, so the UI can mark the pane dead and offer respawn.
- **Hard rule:** the output Channel carries terminal bytes verbatim — Rust never injects logs into it. Rust's own logging goes to stderr/tracing only.

## Workspace persistence (JSON, not SQLite — for now)
Persist layout + pane intent, never live scrollback. Three serialized things in the app-config dir:
- **`workspaces.json`** — the open rail: each Workspace's name, working folder (`cwd`), `LayoutNode` tree, and per-pane `PaneSpec` (command, **launch** cwd, env, title). On launch, rebuild each tree and **respawn** each pane's command in its **launch cwd** — the Workspace folder or the pane's override, *never* the live wandered cwd (Loom does not read `/proc/<pid>/cwd`; panes stay opaque per ADR-0001). Terminals are not restored to prior output — that's ephemeral.
- **`recents.json`** — recently opened working folders + their remembered terminal count, for the wizard's Start step.
- **`presets.json`** — saved one-click configs (name + folder + layout + per-pane commands).

SQLite is deferred to a later milestone only if searchable scrollback / session logging becomes a goal; JSON keeps Rust thin and the config human-inspectable.

## Failure handling
**One principle: every failure degrades to a Dead pane with a visible reason — never crash, never block sibling panes.**
- **Best-effort spawn:** panes spawn independently; one failure doesn't stop the others. Rust never panics on a spawn failure — it emits a pane-error event the UI renders as a Dead pane.
- **Command not found / immediate exit:** correct *by default* — a payoff of the login-shell launch (ADR-0004). `$SHELL -lc "claude"` with no `claude` prints `command not found` *into the pane* and exits → an ordinary Dead pane. On top of that we make the case legible rather than crash-like: a wizard pre-flight (`pty_check_command`, via `lib/agentAvailability.ts`) warns "⚠ not installed" on agents whose program isn't on PATH *before* launch, and the Dead pane's exit-127 overlay reads "command not found: …" with an **Open shell instead** button (`clearPaneCommand`) that respawns the pane as a plain shell. Both are advisory — never a launch gate.
- **Missing working folder** (deleted/unmounted on respawn): don't refuse to open — surface a Workspace-level warning and fall back to spawning in `$HOME`, marked with the reason; restart lets you repoint it.
- **OS-level spawn failure** (out of ptys, permissions): Dead pane carrying the OS error string, logged to stderr.

## Window model
**One OS window**, two layout regions for v1: **left workspace rail | center pane grid**. A single webview, one render tree — keeps the store and PTY→view binding simple.

## Out of scope for v1 (explicit no's)
These are deliberate exclusions, each recorded where relevant. Listed here so scope is unambiguous:
- **Agent-awareness** — no output parsing, token counts, or status badges; panes are opaque (ADR-0001). *(Superseded post-v1: agents became first-class via pushed signals — [ADR-0008](docs/adr/0008-agents-first-class-via-self-report.md) — with an opt-in heuristic output-observer as a lossy fallback — [ADR-0011](docs/adr/0011-heuristic-output-observer.md); the engine hot path stays byte-opaque.)*
- **Right-side Browser panel** — dropped for now. The reference app's `localhost`/docs preview is a whole second surface (embedded navigable webview) off the critical path; users alt-tab to a real browser. Revisit post-v1.
- **Detach/reattach daemon** — quitting kills all PTYs; persistence = respawn, not live-process survival (ADR-0002).
- **Multi-window & tear-off panes** — a Pane lives in one Workspace in the one window; multi-monitor/tear-off is a later feature.
- **tmux-style "window" layer** — hierarchy is two levels (Workspace → Pane) only.
- **Live-keystroke broadcast mirror** — broadcast sends discrete messages, not synchronized typing.
- **SQLite / searchable scrollback** — JSON persistence only until a logging need appears.

## Top risks
1. **[HIGHEST] PTY throughput → smooth render in WebKitGTK via Tauri IPC.** A flood (`yes`) can lock the webview main thread or balloon memory. Mitigate: Rust-side coalescing (time+size flush), raw-byte Channel (not JSON events), xterm's own write buffer, canvas renderer, bounded/drop buffers. **Prove in M0 before anything else.**
2. **Resize correctness.** xterm fit cols/rows must stay in lockstep with `pty_resize`, including reflow and rapid drags. Debounce; resize PTY *after* xterm fit; test odd aspect ratios.
3. **Renderer on WebKitGTK.** WebGL may be broken/slow — canvas leads, DOM is the safe fallback, WebGL is experimental opt-in only; M0 validates the pick (ADR-0006).
4. **Thread/handle scaling.** One reader thread per PTY × many panes. Bound buffers, throttle hidden Workspaces, reap exited children promptly; revisit async/epoll only if thread count bites.
5. **Input latency.** Keystroke → command → `pty_write` must feel instant. Keep the command path trivial; never batch keystrokes behind a tick.
6. **Clipboard & keybinding conflicts on Linux/WebKitGTK.** Ctrl-C ambiguity (SIGINT vs copy), primary-selection paste, xterm selection rendering. Budget M5 time for this.

**Verify against the live stack before finalizing M0:** Tauri 2 `Channel` byte-streaming API + payload shape; `portable-pty` master read/write/resize + child-exit reaping on Linux; `@xterm/xterm` 5.x renderer options + fit addon under WebKitGTK; whether the WebGL addon works at all in this webview.
